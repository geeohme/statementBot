# BrightWave Help

An end-to-end AI chatbot demo that helps customers (and customer service representatives) **understand their monthly cable statements**. Built around a fictional cable provider, **BrightWave Cable**, with six sequential bills for a single demo account. The current bill is engineered to show a noticeable price jump driven by stacked promotional discounts expiring — exactly the kind of moment a customer is most likely to pick up the phone or open a chat.

The chatbot uses **Claude Haiku 4.5** via the Anthropic Messages API for multi-turn conversation. The bills themselves are rendered on the fly from XML to PDF, so the chatbot has the underlying data and the customer sees the same statement the chatbot is reading from.

---

## What it does

- **Explains charges** in plain language, citing specific line items, sections, and dollar amounts from the actual statement.
- **Compares bills** across months and surfaces what changed (e.g., "Why did my May bill go up?" → a markdown table showing each line item delta).
- **Lists statements** as a clickable table by month and year, with view-PDF / download links.
- **Pulls up statement PDFs** inline — the bot can drop a `[View May 2026 statement](...)` link anywhere in chat, and a side panel lets the user pick any statement directly.
- **Stays honest** — only cites numbers that exist in the data, and points the user to a live agent or account portal when something is outside its scope.

The chatbot is intentionally **customer-friendly** in tone. It works equally well whether the user is the customer themselves or a CSR copy-pasting responses back to the customer.

## Architecture

```
┌────────────────────────────────────────────────────────────────────────┐
│                                                                        │
│   Browser  ──HTTP──►  nginx (web)  ─/api/*─►  Express (backend) ──►   │
│                          │                          │                  │
│                          │                          ├──► Anthropic API │
│                          │                          │    (Claude Haiku)│
│                          │                          │                  │
│                          └────────────/api/────────►┴──► Flask         │
│                                                          (pdf_renderer)│
│                                                              │         │
│                                                       reads XMLs       │
│                                                       renders PDF      │
│                                                       (WeasyPrint)     │
└────────────────────────────────────────────────────────────────────────┘
```

Three containers in a single Docker Compose stack:

| Service        | Stack                                     | Purpose                                                                |
| -------------- | ----------------------------------------- | ---------------------------------------------------------------------- |
| `web`          | nginx (alpine) + static HTML/CSS/JS       | Serves the SPA and proxies `/api/*` to the backend.                    |
| `backend`      | Node 20 + Express + `@anthropic-ai/sdk`   | Hosts the chat endpoint, assembles the system prompt, proxies PDFs.    |
| `pdf_renderer` | Python 3.12 + Flask + WeasyPrint          | Loads the XML statements, parses them, renders PDFs via HTML template. |

Statements live as **XML files** (one per month) in `./statements/`. They're generated by a small Python script (`generate_statements.py`) that drives all the math from line items, so the totals always reconcile.

## Quick start

You'll need Docker Desktop and an Anthropic API key.

```bash
cd StatementChatbot
cp .env.example .env
# Edit .env and paste your real key into ANTHROPIC_API_KEY

docker compose up -d
```

Then open **http://localhost:8080**.

To stop:

```bash
docker compose down
```

To rebuild after changing source files (e.g., a prompt fragment or the CSS):

```bash
docker compose build <service>      # backend | pdf_renderer | web
docker compose up -d <service>
```

## How it works

**Statement data.** `generate_statements.py` defines the account, base prices, monthly variations, and the "promo loss" overrides for cycle 6. Section totals and bill totals are computed from line items at runtime — there are no hard-coded sums, so the math always reconciles. Re-running the script regenerates the XML files; the renderer picks them up on next request because it reads them on-demand.

**PDF rendering.** The Flask app loads each XML on startup, parses it into a dictionary, and serves three endpoints per statement:

- `GET /statements/<n>` — raw XML
- `GET /statements/<n>/json` — parsed JSON (used by the backend for chat context)
- `GET /statements/<n>/pdf` — Jinja-rendered HTML run through WeasyPrint into a PDF

The PDF template is a faithful but rebadged take on a typical cable bill — page 1 with the summary and remit stub, page 2 with the detailed charges, taxes, and "What's included" sidebar. When the current bill has promo expirations, an alert callout appears on page 1 and a "What changed this month?" panel appears on page 2.

**Chat backend.** Express exposes `POST /api/chat`, which receives the conversation history from the frontend and forwards it to the Anthropic Messages API. Before every call it rebuilds the **system prompt** by concatenating the seven prompt fragments and appending all six statements as JSON context. The model therefore has the full account in front of it on every turn and can cite any line item with confidence.

The backend also proxies `/api/statements`, `/api/statements/<n>/xml`, and `/api/statements/<n>/pdf` to the renderer, so the frontend only needs to know about one origin.

**Frontend.** Vanilla HTML/CSS/JS, no build step. The left pane is the chat; the right pane is an `<iframe>` of the PDF the user has selected from the picker. Assistant messages are rendered as markdown via `marked.js` and sanitised with `DOMPurify`, so the bot can produce headings, tables, bold, and clickable links to specific statement PDFs. Links to PDFs are automatically opened in a new tab.

The SparkBright theme — a warm amber spark on a deep navy ground — is applied via CSS custom properties at the top of `frontend/styles.css`. The bill itself uses BrightWave's calm teal/blue palette so the printed statement looks like a real cable bill.

## Customising

**Account & data:** Edit `generate_statements.py` to change the customer name, address, services, prices, promo logic, or to add more months. Re-run `python3 generate_statements.py` and restart the renderer.

**Bill look & feel:** Edit `pdf_renderer/templates/statement.html` and `pdf_renderer/static/styles.css`. Rebuild the `pdf_renderer` container.

**Chatbot behaviour:** Edit any file in `backend/prompts/`. Each fragment is focused on one concern (identity, response standards, bill-change patterns, etc.) so it's easy to iterate without breaking the whole prompt. Rebuild the `backend` container.

**Theme:** All chatbot colours are CSS custom properties at the top of `frontend/styles.css`. Update them once and the whole UI follows. Rebuild the `web` container.

**Suggested prompts (chips):** Edit the `<button class="chip">` elements in `frontend/index.html`.

## MVP scope

What's in:

- Bill explanation, line-item Q&A, month-to-month comparison, change explanation
- Standardised response structure with markdown tables
- Clickable PDF links, full PDF viewer, statement picker
- Default-to-most-recent statement
- Customer-friendly tone usable by either customers or CSRs

What's intentionally deferred (and called out in the system prompt so the bot deflects cleanly):

- Admin Q&A content upload
- Reporting / analytics / self-learning from unanswered questions
- Service add/remove/disconnect scenarios (no data for them yet)
- Mobile bundle interactions
